# Архитектура надёжного веб-сервиса

## Принципы

- **Stateless** — сервис не хранит состояние локально, сессии вынесены наружу (Redis).
- **12-factor app** — конфигурация через переменные окружения, логи в stdout, бекграунд-задачи через
  очередь.
- **Idempotent** — повторный запрос не должен дублировать сайд-эффект.
- **Graceful degradation** — при отказе части компонентов сервис продолжает работать в урезанном
  режиме.

## 1. Схема слоёв (снаружи внутрь)

```
Пользователь
    │
    ▼
CDN (CloudFlare / Fastly)
    │
    ▼
DNS → Load Balancer (Nginx / HAProxy / Envoy)
    │
    ▼
API Gateway (Kong / Traefik / Nginx)
    │
    ├─► Auth-service     ──► PostgreSQL (основная БД)
    ├─► Business-service ──► Redis (кеш + сессии)
    ├─► Worker-service   ──► RabbitMQ / NATS (очередь)
    ├─► File-service     ──► S3 (файлы / блобы)
    └─► Search-service   ──► Elasticsearch
            │
            ▼
    Prometheus + Grafana (метрики)
    Loki / ELK           (логи)
    Jaeger / Tempo       (трейсинг)
```

## 2. Компоненты

### 2.1 CDN и балансировщик

- **CDN** — кеширует статику и статические ответы API ближе к пользователю.
- **Load Balancer** (L4/L7) — распределяет трафик на здоровые экземпляры, держит keepalive,
  терминирует TLS.

### 2.2 API Gateway

- Единая точка входа, роутинг запросов к микросервисам.
- Rate limiting, аутентификация (JWT / OAuth2), CORS.
- Request/response трансформация и логирование.

### 2.3 Сервисы (микросервисы, но без фанатизма)

| Сервис | Зона ответственности | Хранилище |
|-----------------|----------------------------------|----------------| | Auth | Регистрация, логин,
токены | PostgreSQL | | Business | Основная бизнес-логика | PostgreSQL | | Worker | Асинхронные
задачи (email, отчёты) | RabbitMQ/Redis | | File | Загрузка/раздача файлов | S3 | | Search |
Полнотекстовый поиск | Elasticsearch |

- **Правило деления**: если модуль имеет свою БД, свой SLA и свою команду — это сервис. Иначе —
  модуль внутри монолита.
- REST/gRPC для синхронных вызовов, очередь — для асинхронных.

### 2.4 Хранилища

| Тип | Технология | Для чего | |---------------|-----------------|------------------------------| |
Реляционная | PostgreSQL | Основные данные, транзакции | | Кеш | Redis | Горячие данные, сессии | |
Поиск | Elasticsearch | Полнотекстовый поиск | | Blob | S3 | Файлы, бэкапы, артефакты | | Очередь |
RabbitMQ / NATS | Асинхронные задачи |

## 3. Надёжность

### 3.1 Отказоустойчивость

- **Redundancy** — минимум 2 экземпляра каждого сервиса, размещённых в разных AZ.
- **Health-check** — k8s liveness/readiness probes для автоматического рестарта и вывода из
  балансировки.
- **Circuit Breaker** — при деградации внешнего сервиса запросы не доходят до него и быстро
  фейлятся, не блокируя вызывающий код.
- **Retry + Backoff** — экспоненциальный backoff с jitter для временных сетевых сбоев.
- **Idempotency key** — клиент передаёт ключ идемпотентности, сервер игнорирует дубликаты.

### 3.2 Консистентность данных

- **Saga** — распределённая транзакция через серию локальных транзакций с компенсациями.
- **Outbox pattern** — изменение БД + событие в очередь атомарно в одной транзакции.
- **Eventual consistency** — допускаем временные расхождения и сводим их фоновым процессом.

### 3.3 Бэкапы и DR

- Ежедневные бэкапы БД, хранить в S3 (минимум 30 дней).
- Процедура восстановления описана в runbook и протестирована.
- Инфраструктура как код (Terraform / Ansible) — кластер можно поднять с нуля.

## 4. Масштабирование

### 4.1 Горизонтальное

- **Stateless-сервисы** — просто добавляем реплики, балансировщик раздаёт трафик.
- **БД** — read-реплики для чтения, primary для записи.
- **Шардинг** (только когда реплик уже недостаточно) — деление по tenant_id или хешу ключа.

### 4.2 Кеширование

| Уровень | Стратегия | |---------------|--------------------------| | CDN | Статика, неизменяемые
ответы API | | In-memory | Локальный кеш (малый TTL) | | Redis | Горячие данные, сессии |

- Инвалидация: TTL + явный инвалидирующий запрос при изменении данных.
- Паттерн **Cache-Aside**: код сам проверяет кеш → нет → читает БД → кладёт в кеш.

## 5. Наблюдаемость

### Три столпа

| Столп | Инструменты | Что даёт |
|-----------|-------------------------------|----------------------------------| | **Logs** | Loki /
ELK | Отладка, аудит, расследование | | **Metrics** | Prometheus + Grafana | SLO, алерты, тренды | |
**Traces** | Jaeger / Tempo | Латенси по эндпоинтам, узкие места |

### Алерты

- Только actionable: латенси > p95 SLO, error rate > порог, диск > 80%, очередь растёт.
- Алерты по четырём золотым сигналам: latency, traffic, errors, saturation.

## 6. CI/CD и инфраструктура

### Пайплайн

```
Git push → CI (линтеры + тесты + security scan)
    │
    ├─► Сборка Docker-образа, пуш в registry
    ├─► Деплой в staging, smoke-тесты
    └─► Деплой в production (canary → rolling)
```

### Стратегии деплоя

| Стратегия | Когда использовать | Риск | |-------------|-------------------------------|------| |
Rolling | Обычный деплой, обратная совместимость | Низкий | | Blue-Green | Критические обновления
схемы БД | Средний | | Canary | Проверка на части трафика | Низкий |

### Инфраструктура как код

- **Terraform** — облачные ресурсы (БД, очереди, DNS).
- **Kubernetes манифесты / Helm** — деплой сервисов.
- Всё в Git, изменения через PR с ревью.

## 7. Безопасность (минимальный набор)

- TLS между всеми компонентами.
- Принцип наименьших привилегий для сервисных аккаунтов.
- Секреты — через HashiCorp Vault или sealed secrets, **никогда не в Git**.
- OWASP Top-10: санитизация ввода, защита от CSRF/XSS.

## 8. Чек-лист «сервис готов к проду»

- [ ] Health-check эндпоинты (`/healthz`, `/readyz`) возвращают корректный статус.
- [ ] Метрики и алерты настроены (latency, errors).
- [ ] Логи в stdout, собираются централизованно.
- [ ] Graceful shutdown (SIGTERM → дообработка текущих запросов → выход).
- [ ] Бэкапы настроены, процедура restore проверена.
- [ ] Runbook написан (типовые инциденты и действия).
- [ ] Rate limiting и circuit breaker включены.
- [ ] CI/CD пайплайн деплоит без ручных действий.
