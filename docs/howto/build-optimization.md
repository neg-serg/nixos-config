# Build Performance Analysis

## Current Bottlenecks

| Фактор | Влияние | Что делать |
|--------|---------|------------|
| **`builtins.pathExists` с абсолютными путями** | Медленная оценка (impure), ломает eval cache | Заменено на feature flag в `session/chat.nix` |
| **Полный профиль (`profile = "full"`)** | Все фичи включены → больше сборок | Использовать `features.devSpeed.enable` для быстрой итерации |
| **`cores = 2`** | Каждый build получает 2 ядра, с 32/16 потоками можно больше | Поднять до `cores = 4` (меньше параллельных задач = быстрее каждая по отдельности) |
| **`max-jobs = 16`** | Много параллельных сборок жрут память | Уменьшить до `max-jobs = 8` для десктопа с 32ГБ ОЗУ — меньше OOM |
| **Нет remote builder** | Всё собирается локально, под нагрузкой CPU простаивает | Настроить `builders` на отдельную машину или VPS |
| **CA derivations (`ca-derivations`)** | Content-addressed — пересборка при изменении зависимости не затрагивает downstream (ещё не стабильно) | Оставить включённым — уже даёт выигрыш |
| **`builtins.readFile` в modules/media.nix** | Чтение файлов при каждой eval → IFD | Заменить на `lib.cleanSourceWith` или path-зависимости |
| **Image-based: `lib.cleanSource` c filter** | Полная переоценка src при каждой сборке | Использовать `lib.cleanSourceWith` с именованными source'ами |
| **Зависимости от nixpkgs-unstable** | Частые обновления → пересборка кеша | Закрепить стабильную версию для base system, overlay для отдельных пакетов |

## Архитектурные проблемы

1. **Всё в одном closure** — systemPackages мешает быстрое и тяжёлое (rustc, gcc) с базовым системным софтом. Любое изменение → перелинковка system-path.

2. **Нет разделения на базу и пользовательское окружение** — home-manager или `nix profile` позволили бы пересобирать только пользовательский слой отдельно от системы.

3. **Большое кол-во импортов в `modules/flat.nix`** — каждый import добавляет время оценки. Не критично, но суммарно ~200-300ms.

## Рекомендации

| Действие | Ожидаемый эффект | Сложность |
|----------|------------------|-----------|
| Поднять `cores = 4` | -15-20% времени сборки | Низкая |
| Включить `devSpeed` для работы над конфигом | Пропустить тяжелые фичи при тестовой сборке | Низкая |
| Вынести `pathExists` в CI/commit-time | Ускорение eval на ~100ms | Средняя |
| Разделить systemPackages на базовые + тяжёлые (через feature-флаги) | Меньше пересборок при изменении софта | Средняя |
| Добавить remote builder | Сбросить нагрузку с десктопа | Высокая |

## Обоснование

- `builtins.pathExists` с абсолютными путями вне репозитория — impure-операция.
  Относительные пути внутри source tree nix кэширует нормально (они детерминированы).
  Единственный impure вызов был в `session/chat.nix` — заменён на feature flag.

- `cores = 2` vs `cores = 4`: nix запускает `max-jobs` параллельных сборок, каждая с `cores`
  потоками. Если у вас 16 physical cores / 32 threads:
  - cores=2, max-jobs=16 → 16 параллельных задач × 2 потока = 32 потока (полная утилизация)
  - cores=4, max-jobs=8 → 8 параллельных задач × 4 потока = 32 потока (та же утилизация, но
    каждая задача выполняется быстрее за счёт меньшего contention на shared cache)
  - cores=2 → больше накладных расходов на IPC между build-процессами

- `profile = "full"` включает все feature-флаги. `devSpeed = true` выключает web.tools,
  floorp (уже отключён), fastfox, qt, fun, ai, torrent — остаётся минимум для теста.

- Remote builder (`--builders`) переносит сборку тяжёлых derivation'ов (rustc, gcc, webkitgtk)
  на другую машину. Локально собирается только финальная линковка system closure.
